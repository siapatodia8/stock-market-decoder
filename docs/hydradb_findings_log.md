### 1. `synthesis_context` is only populated in `mode="thinking"` — `fast` and `auto` return `None` even with `graph_context=True`
- **Method**: SDK
- **Claimed**: `graph_context: true` is documented as the flag controlling whether graph context is returned; `mode` is documented as a recall-strategy selector, not a field-population gate.
- **Found**: Same tenant/query/`graph_context=True`, only `mode` varied: `"fast"` → `None`; `"auto"` → `None`; `"thinking"` → real text. Ruled out result-count truncation (`max_results=20` gave the same result).
- **Where**: `test.ipynb`, tenant `sia-test`.
- **Severity**: Moderate. `mode="thinking"` being necessary doesn't mean it's sufficient — see finding #9.

### 2. Deprecation notice on `client.databases.create` points at fields that don't exist in the current SDK signature
- **Method**: SDK
- **Found**: `client.databases.create(...)` succeeded but returned `meta.deprecation` telling callers to migrate from `tenant_metadata`/`tenant_metadata_schema` to `database_metadata`/`database_metadata_schema` — neither of which exists in the method's actual signature (`inspect.signature` confirmed).
- **Where**: `client.databases.create` response `meta.deprecation`, tenant `stock-market-decoder`.
- **Severity**: Moderate — cosmetic today, but a forward-looking trap: complying with the notice right now would raise a `TypeError` for an unknown kwarg.

### 3. SDK's `document_metadata` requires a JSON array even for a single-document call; error reveals the field is really called `file_metadata` server-side
- **Method**: SDK
- **Found**: `client.context.ingest(..., document_metadata=json.dumps({...}))` (single object) failed: `400 INVALID_INPUT`, `"cannot unmarshal object into Go value of type []json.RawMessage"`. Wrapping in a list (`json.dumps([{...}])`) is required even though the call only ever ingests one document.
- **Where**: `client.context.ingest`, tenant `stock-market-decoder`.
- **Severity**: Moderate — unintuitive given the singular `documents` param; only discoverable via a Go-level error naming an undocumented field. Fixed by wrapping in `[...]`.

### 4. `client.context.status` silently returns `FILE_NOT_FOUND` for real, indexed documents unless `sub_tenant_id` is passed explicitly
- **Method**: SDK
- **Found**: `client.context.status(tenant_id=..., ids=[...])` with no `sub_tenant_id` returned `FILE_NOT_FOUND` for all 13 real, indexed document ids (verified against the dashboard). Adding `sub_tenant_id="default"` (matching ingest-time value) immediately found them.
- **Where**: `client.context.status`, tenant `stock-market-decoder`.
- **Severity**: Significant — an optional param defaulting to `None` behaves as "search an empty scope," not "use the default sub-tenant."

### 5. E6001 ("Storage" failure) is a transient, self-resolving retry issue, not a schema or ingestion-formation problem
- **Method**: SDK
- **Claimed**: Docs state documents fully process in 1–5 minutes; `errored` is a documented terminal status, implying failures surface on a similar timescale and reflect a real, fixable problem with the request.
- **Found**: During SDK ingestion of `stock-market-decoder`, documents intermittently surfaced `error_code: "E6001"`, `"We couldn't save your indexed content"` on the dashboard's Pipeline tab at the Storage stage, sitting there for several minutes before resolving. A `nullable: true` tweak on `doc_summary` (prompted by the deprecation-notice wording in finding #2) was never actually the deciding factor: 12 of 13 documents that hit this error succeeded anyway, purely via HydraDB's own automatic retries, with no schema or request change at all.
- **Where**: Dashboard Pipeline tab, tenant `stock-market-decoder`.
- **Severity**: Moderate — the multi-minute stall before self-resolution could exceed a naive integration's polling timeout, but the underlying cause looks like an async propagation race (document metadata write vs. chunk-level vector insert), not a genuine permanent failure. Recommendation: treat E6001 on Storage as something to tolerate with patience (retries measured in minutes), not something requiring a schema or payload change.

### 6. SDK ingest ids are deterministic (derived from filename + tenant/sub-tenant), not randomly assigned per call
- **Method**: SDK
- **Claimed**: `client.context.ingest()` has no explicit `id`/`file_id` parameter, implying no built-in upsert key unless one is supplied.
- **Found**: A removed document, re-ingested as a fresh call under the same filename, returned the byte-for-byte identical `id` it had before removal. A random/server-generated id could not reproduce this by chance.
- **Where**: `scripts/setup_and_ingest_sdk.py --step ingest --doc-id ...`, tenant `stock-market-decoder`.
- **Severity**: Minor/informational — there is an effective, implicit upsert key (derived from filename + tenant/sub-tenant) even though it isn't an explicit parameter or documented anywhere.

### 7. Re-ingesting under the same (deterministic) id shortly after deleting the original silently fails to materialize — distinct from the E6001 pattern
- **Method**: SDK
- **Found**: Re-ingesting a just-deleted document under its original filename returned `success: true` with the same id as before, but never appeared in the dashboard's Context list, even after several minutes. Re-ingesting the same content under a new filename (new deterministic id) worked normally — appeared within ~6.5s, then went through the same E6001/Vectorisation pattern as the other 12 documents, eventually succeeding.
- **Where**: `scripts/setup_and_ingest_sdk.py`, tenant `stock-market-decoder`.
- **Severity**: Significant — worse failure mode than E6001 since there's no visible error, just a document that never shows up. Workaround: never reuse a just-deleted document's filename/id.
- **Note**: Across all 13 documents, observed time-to-successful-Storage (via automatic retries, no manual fix) ranged ~4.1 to 27.5 minutes. The same document was the slowest case twice in a row, under two different ids — suggests some documents are consistently slower to embed/store, not pure random variance.

### 8. `client.query` also silently returns nothing without explicit `sub_tenant_id` — same trap as #4, in a second SDK method
- **Method**: SDK
- **Found**: `client.query(tenant_id=..., query=..., mode="thinking", graph_context=True)` with no `sub_tenant_id` returned `chunks: []`, `sources: []`, `query_paths: []`, despite the relevant documents confirmed `indexing_status: "completed"`.
- **Where**: `client.query`, tenant `stock-market-decoder`.
- **Severity**: Significant, same reasoning as #4.
- **Resolved**: Adding `sub_tenant_id="default"` fixed it — 13 relevant chunks returned, correctly ranked, plus populated `chunk_relations`/`query_paths`.

### 9. `mode="thinking"`'s `synthesis_context` field does not reliably produce a real answer on this tenant's data
- **Method**: SDK
- **Found**: Across 6 live query attempts (3 distinct phrasings — compound, single-part date-scoped, direct factual with no date/filter language), `synthesis_context` never once produced a grounded answer: 4/6 returned a stub describing the query instead of answering it (e.g. *"The query asks for two distinct types of information... requiring a search filtered by date."*), 2/6 returned a clean `None`. This happened even with strong retrieval (up to 18 `graph_paths`, 11–13 accurate chunks) and even though `chunk_relations`/`query_paths`'s own `combined_context` fields held correct, well-grounded summaries of the same content. A separate cross-check on `sia-test` (2 clean synthetic documents, single narrative) showed the same pattern: one phrasing produced real (if incomplete — it omitted the actual causal fact present in the graph) synthesized text; a differently-worded question against the identical two documents produced the same query-paraphrase stub.
- **Where**: `client.query`, tenant `stock-market-decoder` (`outputs/_ingestion_results_sdk.json`) and tenant `sia-test` (`test.ipynb`).
- **Severity**: Real and reproducible, but not a broken promise — HydraDB's docs never claim `/query` returns a final answer; every official cookbook generates the answer via the caller's own LLM call over `chunks`/`graph_context`, not via `synthesis_context`. The actual issue is discoverability: `synthesis_context` sits in the response schema indistinguishable from `chunk_relations`/`query_paths`, but per its own OpenAPI field description, it's only populated for "multi-step queries with `requires_synthesis=True`" — a flag that isn't exposed as a request parameter anywhere in the public API. Full pipeline/architecture writeup in `CONTEXT_UPDATES.md`.
- **Assumption**: Our own build should not treat `synthesis_context` as the answer source. Detect null/stub and construct the answer from `chunk_relations`/`query_paths`'s `combined_context`, or — matching every official cookbook's pattern — call our own LLM over the formatted retrieval context.

### 10. `document_metadata` silently accepts the wrong shape with no validation error, resulting in permanently empty chunk metadata
- **Method**: SDK
- **Claimed**: N/A — self-diagnosed after `metadata_filters` returned zero results for every shape tried, before assuming it was a HydraDB bug.
- **Found**: Our own SDK ingestion sent `document_metadata` as a flat object (`{"doc_type": ..., "filing_date": ..., ...}`) with no `"id"`/`"metadata"` wrapper — the documented shape (`api-reference/v2/endpoint/ingest-context.md`) requires `[{"id": ..., "metadata": {...fields...}}]`. Ingestion returned `success: true` with no error or warning for the malformed shape, but every chunk's `metadata`/`additional_metadata` came back completely empty at query time — confirmed by inspecting raw chunk fields directly.
- **Where**: `client.context.ingest`, tenant `stock-market-decoder`.
- **Severity**: Moderate on HydraDB's side — a structurally-wrong-but-valid-JSON payload silently discards the metadata instead of failing validation or warning. Was Significant on our side until diagnosed, since it made every `metadata_filters` attempt look broken.
- **Resolved**: fixed the shape, re-ingested. `metadata_filters` now works exactly as documented — top-level plain key-value pairs matching `tenant_metadata_schema` (e.g. `{"filing_date": "2020-12-21"}`), confirmed for both `filing_date` and `doc_type`. A single value wrapped in a list also works (e.g. `{"filing_date": ["2020-12-21"]}`). A list of multiple *different* values does not — tested `{"filing_date": ["2022-02-05", "2022-02-08"]}` (two real dates, each individually confirmed working) and got zero chunks, vs. 4 and 3 chunks respectively for the same two dates queried separately. No "any of" / IN semantics — exact single-value match only. `timeline.py` needs one filtered call per date, merged in code, for any period with more than one real filing date.

### 11. Supplying an explicit `id` in `document_metadata` for a filename previously ingested without one creates a duplicate instead of upserting over it
- **Method**: SDK
- **Claimed**: Implicit — `upsert: true` plus deterministic filename-based ids (finding #6) should mean re-ingesting the same filename always updates the same record.
- **Found**: After fixing finding #10's metadata shape (which added an `"id"` field to `document_metadata` for the first time), re-ingesting all 13 documents did not update the existing records — it created 13 new ones, confirmed via the dashboard (26 of 26 files). The two copies of the same filename had unrelated ids: the original had an auto-assigned hash id (e.g. `fbedf8ab5ec47eba63b752842cb05d5e`), the new one had the caller-supplied id (e.g. `peloton_2022-02-08_8k`). Supplying an id sets the document's identity directly rather than being matched against whatever identity that filename already had.
- **Where**: `client.context.ingest` / `client.context.list`, tenant `stock-market-decoder`.
- **Severity**: Significant — a real trap for anyone incrementally adopting the documented `document_metadata` shape on an already-ingested tenant: correctly adding `"id"` for the first time causes silent duplication, not an update, with no error to react to.
- **Resolved**: identified and removed the 13 stale, empty-metadata copies via `client.context.list` + `client.context.delete`, keeping the 13 correctly-metadata'd ones. Not an ongoing risk going forward — now that `id` is supplied consistently every time, future re-ingestion should upsert correctly.

### 12. `mode="thinking"`'s internal reranking is measurably non-deterministic near the `max_results` boundary — a correct, relevant chunk can silently vanish across identical repeated calls
- **Method**: SDK
- **Claimed**: `essentials/v2/api-results.md`'s practical-guidance section frames `max_results` purely as a token-budget/downstream-reranking knob — *"Start small on chunks. `max_results: 10` is a reasonable default... raise to 20 if you rerank downstream."* No mention of a correctness or stability risk at low values.
- **Found**: Ran the identical `mode="thinking", max_results=10` call (`metadata_filters={"filing_date": "2022-02-05"}`) 8 times in a row. The correct source document (`peloton_2022-02-05_8k.md`) was present in only 2/8 runs, and its `relevancy_score` swung wildly across those hits (0.044 and 0.881 — not close values, a 20x spread). Raising only `max_results` to 20 (same `mode="thinking"` call, nothing else changed) fixed it 8/8, with the score stabilizing in a tight band (~0.847–0.881). Separately, switching to `mode="fast"` at the original `max_results=10` was also 8/8 stable, with the *identical* `relevancy_score` every single run (`1.4244978427886963`). This isolates the cause specifically to `thinking` mode's internal reranking step, not the underlying retrieval/embedding step, which is itself perfectly deterministic.
- **Where**: `tests/test_chunk_retrieval_stability.py`, `outputs/_chunk_retrieval_stability_results.json`, tenant `stock-market-decoder`.
- **Severity**: Significant, and not merely theoretical — this caused a real, silently-wrong production output. Two consecutive `timeline.py` runs mislabeled Feb 5–7 disclosures as "February 8" because the correct 8-K vanished from the result set and only a press release (whose own byline reads "February 8, 2022" despite describing earlier events) survived. This is a correctness failure, not a token-budget inconvenience — the docs' framing of `max_results` doesn't warn a developer this could happen.
- **Assumption/resolution**: fixed via `max_results=20` for `timeline.py`'s per-date queries specifically. Not proven this fully generalizes past this dataset's scale (13 documents, single tenant) — larger corpora may need a higher threshold, or may still see boundary effects at 20. Flagging as an open risk rather than a closed one.

### 13. Chunk-level metadata field is exposed as `.metadata` on the SDK's Python object but named `additional_metadata` in the docs' own JSON schema example for the identical field
- **Method**: SDK, cross-checked against `docs.hydradb.com/essentials/v2/api-results.md`
- **Claimed**: That page's response-shape example, and its "Full flow without SDK" reference implementation, both name this field `additional_metadata` on each chunk (e.g. `chunk.get("chunk_content", "")` / `additional_metadata: {"author": "Support Team"}` in the schema sample).
- **Found**: `backend/synthesis.py` accesses the same field via `getattr(chunk, "metadata", None)` — and it works, producing correct, chunk-specific values in production (`outputs/timeline_cache.json`'s `doc_summaries`, e.g. *"Peloton announces agreement to acquire Precor for $420M to establish U.S. manufacturing capacity"* correctly paired with the 8-K chunk, and a distinct summary correctly paired with the press-release chunk). This confirms the SDK's Python chunk object genuinely exposes the field as `.metadata`, not `.additional_metadata` — diverging from the docs' own schema example for that same field.
- **Where**: `backend/synthesis.py` (`get_context_snippets`/`doc_summaries` logic), tenant `stock-market-decoder`.
- **Severity**: Minor — a small, self-contained doc-accuracy gap; the SDK's object shape doesn't mirror the raw JSON shown on the same docs page that documents its own usage.

### 14. Cross-document entity resolution (Tier 4 claim) does not reliably merge the same real-world entity across documents, including on the primary use case the claim's own pilot test was built to cover
- **Method**: SDK + Dashboard UI
- **Claimed**: `01_hydradb_product_and_claims.md`'s Tier 4 claims list, verbatim: *"Entity resolution: recognizes when two differently-worded mentions refer to the same real-world entity."* A pre-project pilot test (two synthetic connected PDFs referencing the same "Project Atlas" entity under different phrasing) confirmed this worked correctly at small scale — 19 merged entities in the combined graph vs. 25 if counted per-document separately, with the shared entity collapsed to one node, not duplicated. `client.context.relations()` returns each entity with a distinct `entity_id` (not just a display-name string), meaning entity identity is something HydraDB resolves internally at extraction time, not left to the caller.
- **Found**: Pulled `context.relations()` for all 5 real documents cited by one real multi-document event (Feb 2022 CEO transition + restructuring: two 8-Ks, a board press release, a restructuring press release, and a shareholder letter), merged the 63 returned triplets, and reviewed all 56 resulting unique entity names by hand. Independently cross-checked the two clearest cases directly in the live HydraDB dashboard's own Graph view (`dashboard.hydradb.com/graph`) — not just our own SDK consumption of it — and the same splits are visible there too.
  - **Correctly merged (exact-name match across documents, working as claimed)**: Barry McCarthy (3 docs), John Foley (4 docs), Angel L. Mendez (3 docs), Erik Blachford (3 docs), Jonathan Mildenhall (3 docs), Spotify (2 docs), William Lynch (2 docs), the "non-executive director" role (2 docs), "Peloton Output Park" (2 docs), "workforce reduction" (2 docs), and "restructuring initiatives" (2 docs) — 11 entities in this one event alone, each collapsed to a single node with edges from every document that mentioned it, exactly as claimed.
  - **Correctly kept separate (different real-world entities that happen to share a word — good precision, not a resolution failure)**: "Peloton" (the company) vs. "Peloton Board of Directors" (a distinct governing body) vs. "Peloton Guide" (a product) vs. "Peloton Output Park" (a physical facility/project) vs. "President of Peloton" (a role title). The live dashboard's own node-search autocomplete independently confirms this same set of distinct entities tenant-wide (searching "peloton" surfaces "peloton output park," "peloton tread," "peloton corporate wellness program," "peloton bike," "peloton interactive, inc.," "peloton board of directors," and "president of peloton" as separate nodes) — HydraDB is not naively keyword-matching "Peloton," it genuinely distinguishes sub-entities correctly.
  - **Should have merged but didn't**: "peloton" and "peloton interactive, inc." are the same company, split into two nodes with two different `entity_id`s (`67d93c...` vs. `d8f229...`) — and this is the hub of the entire event graph, so the split effectively cuts the combined graph's connectivity in half instead of unifying it under one node. Confirmed independently in the live dashboard: searching "peloton" there returns both "peloton" and "peloton interactive, inc." as separate autocomplete entries, and the rendered graph shows "peloton interactive, inc." as a large, heavily-connected hub node while "peloton" does not appear merged into it. Same pattern, smaller scale, also dashboard-confirmed: "netflix" and "netflix, inc." are the same company (both describing Barry McCarthy's Netflix tenure — one 8-K uses the formal name, the shareholder letter uses the short form), returned as two separate autocomplete entries when searching "netfli" in the dashboard. A third, less clean-cut case from the API data alone (not independently re-verified in the dashboard in this pass): John Foley's post-transition title is extracted as up to four separate role nodes across documents — "executive chair" (the dominant form, 3 docs), "executive chair of the board of directors" (1 doc), "executive chairman of the board" (1 doc), and "co-founder and chair" (1 doc) — fragmenting one person's one real title across four nodes instead of one.
- **Where**: `tests/test_knowledge_graph_relations.py`, `outputs/_knowledge_graph_relations_test_results.json`, tenant `stock-market-decoder`; cross-checked against `dashboard.hydradb.com/graph` for the `stock-market-decoder`/`default` tenant.
- **Severity**: Significant — this is a direct, reproducible counterexample to a specifically-tested Tier 4 claim, on the exact scenario (cross-document, same real-world entity, different documents each using their own naming convention) the claim's own pre-project pilot test was designed to validate, and it reproduces on HydraDB's own primary dashboard graph view, not just on data we constructed. The pilot test's synthetic two-document "Project Atlas" case was clean (one obviously-identical entity name, no legal-name-vs-shorthand variation); real SEC filings and press releases routinely refer to the same company by its formal legal name in some documents and a shorthand in others, which is exactly the case this resolution gap falls down on.
- **No source-level fix available**: Checked the full SDK surface before deciding how to handle this. The `context` client exposes exactly seven methods — `delete`, `ingest`, `inspect`, `list`, `relations`, `update_source_metadata`, `status` — and none of them edit, merge, or alias a graph entity. `update_source_metadata` only edits metadata on a *source* (document), not a node. Entity identity (and its `entity_id`) is assigned once, at ingestion-time extraction, with no exposed way to correct it after the fact. The only source-level lever at all would be deleting and re-ingesting documents with more consistent entity phrasing baked into the source text — a real intervention with real risk (finding #11 already showed re-ingestion with explicit IDs can silently duplicate instead of upsert if done wrong), and not even guaranteed to fix the resolution gap, since this finding shows HydraDB doesn't reliably merge legal-name-vs-shorthand variants even when the "same-ness" is obvious. Not pursued for that reason.
- **Note on propagation, for context**: every event's merged knowledge graph in this app is built from a live `context.relations()` call against this same underlying tenant graph at request time — it is not a separate cached copy. So a true source-level fix, had one been available, would have propagated to every event's graph automatically with no extra app-layer work. That's exactly why fixing at the source would have been the "right" place to do this — it just isn't exposed to us, which is what pushes the fix down to the app layer instead.
- **Status**: Resolved via app-layer workaround — case-insensitive exact-name matching (already free, merges 11+ entities correctly per event with no extra code) plus a small manual alias table for the confirmed gaps (`peloton` ↔ `peloton interactive, inc.`, `netflix` ↔ `netflix, inc.`, and the "executive chair" role-title cluster), applied only at knowledge-graph-build time, per request, since there is nowhere upstream to fix it once. Does not touch `synthesis.py`, `timeline.py`, or the existing narrative/evidence pipeline — additive to the new knowledge-graph feature only.

### 15. Undocumented `_retrieval_source` field surfaces in chunk `additional_metadata`, unrelated to caller-supplied metadata, even with `graph_context=False`
- **Method**: SDK
- **Claimed**: `essentials/v2/api-results.md`'s chunk response-shape schema documents `additional_metadata` as the bucket for caller-supplied custom fields (e.g. `{"author": "Support Team"}`), with no mention of reserved/internal keys.
- **Found**: An unfiltered `client.query(..., mode="thinking", graph_context=False, max_results=5)` call returned 8 chunks; 6 had `additional_metadata: null` as expected (this project never sets any), but 2 chunks (both from `peloton_2021-08-26_shareholder-letter_v2.md`) had `additional_metadata: {"_retrieval_source": "graph"}` instead — a key never supplied by our ingestion code and not documented anywhere. This occurred with `graph_context=False` on the call, suggesting the underlying retrieval strategy can involve the graph regardless of that caller-facing flag, and that HydraDB tags individual chunks with their retrieval path internally.
- **Where**: `tests/test_metadata_filter_bucket.py`, `outputs/_metadata_filter_diagnostic_results.json`, tenant `stock-decoder`.
- **Severity**: Minor — doesn't break anything in this project (nothing reads `additional_metadata` downstream, per finding #13), but it's an undocumented, reserved-looking key on a schema the docs describe as caller-owned, and raises a real question about what `graph_context=False` actually suppresses.
- **Not pursued further**: single observation, not re-run repeatedly — low-stakes documentation gap rather than a correctness issue.
- **Provenance note**: not new behavior — confirmed via `git show e2eb109:outputs/_metadata_filter_diagnostic_results.json` that the original build's run of this same diagnostic already captured the identical `{"_retrieval_source": "graph"}` tag (on 2 different chunks that time: the Feb 2022 shareholder letter and board press release). The original findings log (`ea6824c`, last commit before this rerun) never mentions it — the raw evidence existed then too, it just wasn't noticed or written up. Reproduces across two independent builds; only newly *documented* now, not newly *occurring*.

### 16. Chunk schema has no offset/position field — no way to locate a retrieved chunk's exact span within its source document
- **Method**: SDK (source code inspection, not a live API call)
- **Claimed**: N/A — no explicit claim either way in the docs; this surfaced while designing a "cite and highlight the exact passage" UI feature, a common expectation for any RAG-style app built on top of a retrieval API.
- **Found**: Read `SearchV2Chunk`'s full field list directly from the installed SDK (`hydra_db/types/search_v2chunk.py`): `additional_metadata`, `chunk_content`, `chunk_uuid`, `extra_context_ids`, `id`, `layout`, `metadata`, `relevancy_score`, `source_last_updated_time`, `source_title`, `source_type`, `source_upload_time`, `sub_tenant_id`. No field anywhere expresses a character offset, page number, line number, or any other pointer to where `chunk_content`'s text sits within the original source document. `client.context.inspect()` was also checked as a possible source of this — it only fetches a source's raw/inferred content and a downloadable URL, not per-chunk position data.
- **Where**: `.venv/lib/python3.13/site-packages/hydra_db/types/search_v2chunk.py`, `hydra_db/context/client.py` (`inspect()`), tenant `stock-decoder`.
- **Severity**: Moderate — not a correctness bug (nothing is wrong or inconsistent), but a real, verified capability gap for a common downstream use case. An app wanting to highlight the exact retrieved passage within a full document viewer (rather than just naming the source document) cannot do so using any HydraDB-returned field — it has to fall back to its own text-matching (e.g. substring/fuzzy match of `chunk_content` against the full document text it already holds), entirely outside HydraDB.
- **Also checked while investigating this**: `query_by` (`hybrid`/`text`) and `operator` (`or`/`and`/`phrase`) are real parameters on `client.query()`, confirmed directly from its signature — but they control *which chunks are relevant to a search query across a corpus* (a retrieval-ranking decision), not *where a known chunk sits inside a known document* (a within-document alignment problem). Re-querying with the chunk's own text and `operator="phrase"` would only return the same already-known `chunk_content` again, still with no offset — not a workaround.
- **Status**: Not pursued as an app fix in this project (would need our own substring/fuzzy-match code against the already-fetched document text, no HydraDB call involved) — logged here since it's a genuine, source-verified gap worth flagging to HydraDB, not something to build around silently.
