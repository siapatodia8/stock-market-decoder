# SDK & Ingestion Findings

Setup, authentication, and document ingestion behavior confirmed hands-on against the live HydraDB Python SDK.

---

| Finding | Expected | Found | Severity |
|---|---|---|---|
| Missing `sub_tenant_id` silently returns empty results | Docs: "If omitted, the default sub-tenant created during tenant setup will be used." | With `sub_tenant_id` omitted, `context.status()` and `query()` returned empty / not-found even though fully-indexed data existed in an explicitly-specified collection - omission resolves to the system default sub-tenant, not to where the account's data lives, and nothing in the response indicates which scope was searched, making the empty result indistinguishable from data loss. Corroborating raw-API finding (separate tenant): no sub-tenant exists at tenant setup - the auto collection appears only on first write, under a generated id - contradicting the docs' "created during tenant setup." | Significant |
| Re-ingesting a just-deleted id silently fails to appear | Re-ingesting under a known, valid id either succeeds visibly or errors clearly | Two instances of the same test, different outcomes. First instance (SDK build): returned `success: true` with the correct id, but the document never appeared - not in queries or the dashboard - and the tenant still shows only the renamed replacement (`_v2`), with the original filename absent. A different, silent failure mode from E6001; workaround at the time was re-ingesting under a new filename. Second instance (raw API, fresh database): did not reproduce - clean reappearance with the same deterministic id in ~30-60s (see the API Findings reconciliation table). Possibly fixed; both observations kept. When it did fail, nothing in any response distinguished "accepted and vanished" from "accepted and processing." | Significant (first instance; not reproduced in second) |
| Deprecation notice references nonexistent fields | A deprecation notice names the fields to migrate to | `client.databases.create()` succeeds but returns a deprecation notice pointing callers to `database_metadata`/`database_metadata_schema` - neither field exists in the method's actual signature. Complying with it today would raise a `TypeError`. | Moderate |
| `document_metadata` payload shape is unforgiving | A malformed `document_metadata` shape is rejected with a designed error naming the public field and the expected shape | Sending a plain object instead of the documented JSON array is rejected with raw parser internals (`json: cannot unmarshal object into Go value of type []json.RawMessage`), referencing `file_metadata` - a name absent from the ingest and error reference pages (docs say `document_metadata`). The error-responses page's own troubleshooting section explicitly lists this exact cause ("sent as an object instead of a JSON-stringified multipart field"), but the live error carries none of that. Beyond that, a valid JSON array whose entries carry the metadata fields at the top level (missing the required `id`/`metadata` wrapper, so no entry key is recognized) is silently accepted with `success: true` - no error, no unrecognized-key warning - leaving every chunk's metadata empty until manually repaired via `PATCH /context/sources/{id}/metadata`. | Moderate |
| E6001 storage errors are transient, not permanent | Docs describe `errored` as a terminal, real failure state | Documents intermittently surfaced `E6001` for several minutes before resolving on their own via automatic retries, with no schema or payload change needed. Worth budgeting patience for, not treating as a hard failure. | Moderate |
| Ingest ids are deterministic | No explicit id parameter suggests no built-in upsert key | A removed-then-re-ingested document under the same filename gets back the exact same id - ids are derived from filename + tenant/sub-tenant, not randomly assigned. | Informational |
